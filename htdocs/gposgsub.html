The GPOS and GSUB opentype tables

These two tables are used for positioning and substituting glyphs. The GPOS table can control things like: kerning, accent positioning, cursive joining, etc. The GSUB table can control things like: ligatures, arabic forms, vertical rotation, conversion to small caps, indic glyph rearangment, etc.

PfaEdit does not support the full range of possibilities inherent in these two tables.

The GPOS table

PfaEdit will read the following sub tables of the GPOS table:
'kern' The kerning sub-table PfaEdit will read kerning information from the GPOS table, also from afm and tfm files and macintoy FOND resources. Users can create and modify kerning with the metrics view.
'mark' Mark Positioning PfaEdit will read anchor points from the GPOS table. I am not aware of any way to provide this information with type1 fonts. Users can create and modify anchor points in the character view.
There are several different tables which specify anchor points which position other characters with respect to the current one. For a description of anchor points character view.
'blwm' Below Baseline Mark Positioning PfaEdit will read anchor points from the GPOS table. I do not know how this sub-table's semantics differ from those of the 'mark' table (the opentype docs say this has something to do with indic languages, but don't describe why 'mark' is inadiquate). PfaEdit treats this just like 'mark'. Users can create and modify anchor points in the character view.
'mkmk' Mark to Mark Positioning PfaEdit will read anchor points from the GPOS table. Users can create and modify anchor points in the character view.

This table is used for positioning marks (diacritic marks) around other diacritic marks.

PfaEdit is also able to create a GPOS table with all the above sub-tables.

The GSUB table

PfaEdit will read the following sub tables of the GSUB table:
'liga' The standard ligature sub-table. PfaEdit will read ligature information from the GPOS table, also from afm and tfm files. Users can create and modify ligatures with the character information dialog.
There are several different tables which specify ligature data. The different tables classify ligatures into groups as:
  • Required ligatures must always be applied
  • Standard ligatures should be applied
  • Discretionary ligatures may be applied
  • Historic ligatures were common at one point but have fallen out of use and in most cases should not be applied.
  • Fractions will take the string "3/4" and replace it with "¾"
'rlig' The required ligature sub-table. PfaEdit will read ligature information from the GPOS table, also from afm and tfm files. Users can create and modify ligatures with the character information dialog.
'dlig' The discretionary ligature sub-table. PfaEdit will read ligature information from the GPOS table, also from afm and tfm files. Users can create and modify ligatures with the character information dialog.
'hlig' The historic ligature sub-table. PfaEdit will read ligature information from the GPOS table, also from afm and tfm files. Users can create and modify ligatures with the character information dialog.
'frac' The fraction ligature sub-table. PfaEdit will read ligature information from the GPOS table, also from afm and tfm files. Users can create and modify ligatures with the character information dialog.

PfaEdit can write all the above subtables of GSUB, and it may also write
'vrt2' The vertical alternates and rotation sub-table This is used when Japanese or Chinese or Korean writing is displayed vertically. Some characters have different varients depending on the writing direction (parentheses for instance), also roman text should be rotated. PfaEdit gathers the information for this table from the names of the characters in the font. If a character is named "vertuniXXXX" then it is assumed to be the vertical varient of "uniXXXX", or if a character is named "vertcid_nnn" it is assumed to be the vertical varient of CID nnn.
'init' The initial forms sub-table PfaEdit will generate this if the font contains arabic characters in the ranges U+0600-U+06ff and U+FB00-UFEFF. This will map each non-formed arabic character (the one in the U+0600 area) to its initial form varient (>U+FB00).
In renaissance writing this sub-table might be used to map short-s to long.
'medi' The medial forms sub-table PfaEdit will generate this if the font contains arabic characters in the ranges U+0600-U+06ff and U+FB00-UFEFF. This will map each non-formed arabic character (the one in the U+0600 area) to its medial form varient (>U+FB00).
In renaissance writing this sub-table might be used to map short-s to long.
'fina' The final forms sub-table PfaEdit will generate this if the font contains arabic characters in the ranges U+0600-U+06ff and U+FB00-UFEFF. This will map each non-formed arabic character (the one in the U+0600 area) to its final form varient (>U+FB00).
In renaissance writing this sub-table might be used to map long-s to short.
'isol' The isolated forms sub-table PfaEdit will generate this if the font contains arabic characters in the ranges U+0600-U+06ff and U+FB00-UFEFF. This will map each non-formed arabic character (the one in the U+0600 area) to its isolated form varient (>U+FB00).
'smcp' The Small Capitals sub-table PfaEdit will generate this if the font contains glyphs with names that end in ".sc" or ".oldstyle". It provides a mapping between "a" and "a.sc" (or between "zero" and "zero.oldstyle". I cannot fathom why Adobe had decided that oldstyle is similar to small caps, but they say that's what this table does.
'c2sc' The Small Capitals from capitals sub-table PfaEdit will generate this if the font contains glyphs with names that end in ".sc". It provides a mapping between "A" and "a.sc".
'rlig' The required ligature sub-table. Any ligatures marked as required will automagically be added to the standard ligature sub-table as well as being output in the required ligature sub-table.
